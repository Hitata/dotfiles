Default to deciding and executing. Skip ceremony. Don't narrate planned edits before tool calls. Don't offer demos or walkthroughs before building.

**Why:** Across many sessions he corrected the same patterns — "Are you asking me 1 by 1? Just decide on your own", "No need to show me demo, just work on it", and found pre-edit bullet-point plans verbose ("the diff speaks for itself"). Process overhead = friction. He decides fast, moves fast.

**How to apply:**
- When the direction is clear, skip multi-proposal brainstorming. Propose alternatives only when genuinely ambiguous.
- Go straight to the tool call for UI edits and other scoped changes — no preamble.
- When presenting real options, use short labeled lists (A/B/C); he replies with a single letter.
- "yes" / "oke" / "go ahead" / "continue" / "gut" = full approval. Skip recaps, move on.
- Don't build a preview demo before the real thing. Build, deploy, let him check.
- Skip invoking `using-superpowers` for routine scoped tasks (UI styling tweaks, single-file edits). Reserve it for complex, multi-step, or ambiguous work.
- When he *does* want planning up front, he says so explicitly ("make it an issue", "plan ahead", "research first").
