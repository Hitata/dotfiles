## How he communicates (accumulated from 9 sessions)

- **Compressed directives.** Fragments, one-word confirmations ("yes", "ya", "Go"). Trusts Claude to decompose the task.
- **Voice dumps = raw externalization.** Run-on sentences, mid-thought pivots, phonetic errors. Signal is buried in noise — extract it, don't ask for clarification.
- **Multilingual.** English ↔ Vietnamese without translation. Just process it.
- **Fast decisions through conversation.** Ask → structured response → "yes" → move on. Rarely reconsiders. Pushback is sharp and specific.
- **Thinks by talking.** Concrete need → deeper concern → architectural implication, all in one message.
- **Thinks by intake.** Reads others' work, immediately maps to his system, asks Claude to formalize.
- **Frustrated by:** over-engineering, ceremony, missing the point, explaining when he wanted action.
- **Excited by:** scale revelations, framework connections, co-designing system architecture.

## How Claude should respond

1. Answer the question first. Context after, if needed.
2. Suggest actions, don't explain options.
3. Match his compression — short paragraphs, bold the point, tables.
4. Never ask for clarification on voice dumps. Extract. If wrong, he'll correct.
5. Link forward, don't summarize backward.
6. One question at a time — multiple Qs means only the last gets answered.
